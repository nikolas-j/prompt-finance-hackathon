Training principles · how the score is learned

The model learns the shape of retrieval

For each question, retrieval leaves a fingerprint - which chunks ranked highest, by how much, and how consistent they are. We learned that this fingerprint alone is enough to predict whether the LLM's answer will be complete - without re-reading the answer or running another model.

The pipeline at a glance

The classifier sits on the retrieval step, not the generation step. Same features the search engine already computes - zero extra LLM calls.

Crucially, the classifier never sees the question text or the answer text - only the numerical shape of retrieval. That makes it cheap to run on every query and stable across domains.

Three ideas, in plain English

1
Retrieval has a fingerprint
When the right answer is in one clear chunk, the top hit beats the rest by a wide margin and the chunks all reference the same statute. When the answer is scattered or missing, similarity scores flatten out and statute references jump around.
2
The fingerprint tracks completeness
On our 83-question evaluation bank, the same fingerprint shapes appear over and over: certain shapes predict a complete answer, others predict a missing detail. A tiny classifier learns the mapping in seconds.
3
Cheap, honest, transferable
No extra LLM call. No re-reading the answer. The same features ship on every query, so the score is always available - and the model can say “may be incomplete” instead of confidently bluffing.

Two questions, two pathways

Both questions are about lahjavero (gift tax). The retrieval fingerprint - not the topic - is what changes the score.

Calibrated on 83 expert-written questions

The classifier is trained and validated on a hand-curated bank covering basic-tier lookups all the way through multi-rule scenarios. An LLM judge labels each answer PASS / FAIL fact by fact; the labels drive 5-fold cross-validation.
